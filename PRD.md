# PRD — Trellis:研究進度管理 Web App

> 版本 1.0|2026-07-21|作者:Sybil|開發方式:VS Code + Claude Code
> Trellis(棚架):支撐植物生長的構造。此 app 支撐 2026 暑假 → 2027 年 7 月所有研究與學習目標。

---

## 1. 產品概述

### 1.1 一句話定義
一個免費、極簡、iPhone 玻璃質感的個人研究進度管理 PWA:手機隨時輸入進度、隨處檢視;桌面端由 Claude Code 自動解析輸入、更新資料、調整時程。

### 1.2 要解決的問題
- 2026 暑假到 2027/7 有多條並行任務線(論文、AI 比賽、原生植物計畫、工具學習、讀書、日文 N1),缺乏單一總覽。
- 口試時間(2027/1 或 2027/7)未定,兩種情境的 schedule 完全不同,需要並行規劃。
- 論文格式規定繁瑣(系上慣例 + 戶外遊憩研究 APA7 + 造園景觀學報),需要可自動整理、最終可逐項比對的 checklist。
- 文獻閱讀與附錄引用原文摘錄需要固定模板與一致性檢查。

### 1.3 使用者
單一使用者(Sybil)。無多人、無權限系統、無註冊登入流程。

### 1.4 成本原則(硬性限制)
- **固定月費 = $0。** 全部使用免費方案:GitHub(repo + Pages + API)。
- 不引入任何訂閱制服務。若未來要自訂網域(選配),為唯一年費項目,預設不買、使用 `*.github.io`。

## 2. 技術架構

### 2.1 總覽:Git repo 即資料庫

```
┌─────────────┐  GitHub Contents API   ┌──────────────────┐
│ iPhone PWA   │ ──────読写──────────▶ │  GitHub Repo      │
│ (輸入+檢視)  │ ◀───靜態頁面─────────  │  data/*.json      │
└─────────────┘                        │  notes/*.md       │
                                       └──────▲───────────┘
┌──────────────────────┐    git pull/push     │
│ 筆電 VS Code +        │ ─────────────────────┘
│ Claude Code(自動化) │  解析 inbox → 更新 data → commit → 自動部署
└──────────────────────┘
```

- **前端**:純靜態 PWA(vanilla JS 或 Vite + 原生 Web Components,不用重框架)。部署於 GitHub Pages,push 即自動更新。
- **資料**:全部是 repo 內的 JSON / Markdown 檔。沒有資料庫、沒有伺服器。
- **手機寫入**:PWA 以 fine-grained Personal Access Token(只授權此 repo 的 Contents 讀寫)呼叫 GitHub Contents API 直接 commit。Token 存在手機瀏覽器 localStorage,首次使用時輸入一次。
- **桌面自動化**:Claude Code 在 repo 內工作,依 `CLAUDE.md` 的規則解析新輸入、更新 JSON、重算時程、commit + push。
- **備份**:git 歷史即完整備份,可隨時回溯任何一天的狀態。
- **離線**:Service Worker 快取,離線可讀;離線輸入暫存,連線後補送。

### 2.2 Repo 結構(同時也是資料歸類方案,見 §8)

```
trellis/
├─ CLAUDE.md              # Claude Code 自動化規則(見 §7)
├─ app/                   # PWA 原始碼(單頁)
├─ data/
│  ├─ projects.json       # 7 個專案定義
│  ├─ tasks.json          # 所有任務
│  ├─ milestones.json     # 里程碑與兩案口試時程
│  ├─ literature.json     # 文獻清單索引(狀態機)
│  ├─ format-checklist.json # 論文格式檢核表(§6)
│  ├─ books.json          # 30 本書清單與進度
│  └─ demo/               # 同結構的 demo 資料一套
├─ inbox/
│  └─ inbox.md            # 快速輸入暫存區(手機/筆電皆可丟)
└─ notes/
   ├─ literature/         # 每篇文獻一個 md(依 §5 模板)
   └─ appendix-quotes/    # 附錄引用原文摘錄,每篇文獻一檔
```

### 2.3 明確不做
- 不做帳號系統、不做多人協作、不做原生 app、不做推播伺服器(提醒用 iOS 行事曆/捷徑替代)、不做後端。

## 3. UI / UX 需求

### 3.1 設計語言
- **iOS 26 Liquid Glass 動態玻璃**:半透明卡片、backdrop-filter 模糊、細邊框高光、大圓角、深淺色自動切換。
- 極簡:預設畫面只有「今天要做的事 + 一個輸入框」。其他資訊往下滑或進分頁才出現。
- 手機優先(iPhone Safari 加入主畫面 = 全螢幕 PWA);桌面版同一份 code,寬螢幕時變多欄。

### 3.2 頁面(共 5 個,不再多)

| # | 頁面 | 內容 |
|---|------|------|
| 1 | **今天** | 今日/本週任務、快速輸入框(置底,玻璃浮條)、各專案進度環 |
| 2 | **時間軸** | 2026/7 → 2027/7 橫向時間軸,7 個專案泳道 + 兩案口試雙軌(§4.2) |
| 3 | **文獻** | 文獻狀態看板(待讀→摘要→詳讀→已引用),點入見模板內容 |
| 4 | **檢核** | 論文格式 checklist(§6),可勾選、顯示完成率,最終比對模式 |
| 5 | **設定** | PAT 設定、Demo 模式開關、資料匯出 |

### 3.3 快速輸入(核心互動)
- 輸入框接受自然語言,例:「倫理課程上完第3章」「讀完《設計中的設計》」「文獻 Chen2024 摘要寫好了」。
- 手機端:直接 append 到 `inbox/inbox.md`(一行一筆,自動加時間戳)並立即在本地 UI 樂觀更新。
- 真正的結構化更新由 Claude Code 在桌面端完成(§7)。**手機不需要懂語意,只負責收進 inbox。**

### 3.4 Demo 模式
- 設定頁一鍵切換,改讀 `data/demo/`,UI 掛「DEMO」浮水印。用於展示、截圖、比賽 demo,不動真實資料。

## 5. 文獻管理模組

### 5.1 狀態機
`待讀 → 已寫摘要 → 已詳讀 → 已引用(需附錄摘錄) `,literature.json 存索引與狀態,內容在 `notes/literature/<citekey>.md`。

### 5.2 文獻筆記模板(每篇一檔,Claude Code 建檔時自動套)

**A. 摘要層(≤200 字,回答三問):**
1. 這篇論文的研究問題是什麼?
2. 這個問題用什麼樣的方法來解決或驗證?
3. 用這個方法解決後,最後得到什麼結果?

**B. 詳細層(十問,標註對應 IMRD 段落):**
1. 研究目的為何?(I)
2. 研究問題為何?(I)
3. 研究問題的背景為何?過去如何解決?引用哪些理論?(I)
4. 主要研究假設與研究架構為何?(M)
5. 作者如何抽樣與獲得資料?(M)
6. 主要變項為何(獨變項、依變項、控制變項)?(M)
7. 使用了哪些統計方法?(M)
8. 有多少重要發現?(R)
9. 結果有何限制?如何應用?(D)
10. 我對這篇論文整體的評論?

**C. 引用意圖:** 我打算在論文哪一節引用、引用原因。

### 5.3 附錄引用摘錄與一致性檢查
- 每一篇「已引用」文獻須在 `notes/appendix-quotes/<citekey>.md` 摘錄**引用的具體原文**。
- Claude Code 定期執行檢查:(a) 論文引用清單 vs 附錄摘錄是否一一對應、無遺漏;(b) 摘錄原文與「引用意圖」是否一致、正確;不一致者列入檢核頁警示。
- 論文完稿時,由這批檔案直接彙整產出附錄。

## 6. 論文格式檢核表(format-checklist.json 完整初始內容)

依「東海大學景觀學系碩士班論文撰寫格式規定」(版面以系上慣例為主)。每項可勾選,支援「最終比對模式」逐項核對。

### 6.1 總則
- [ ] 格式內容依東海大學景觀學系碩士班論文撰寫格式規定
- [ ] 編排:縱向橫寫,以 Microsoft Word 打字
- [ ] 紙張:正文 A4;圖表或地圖以此為原則但不以此為限
- [ ] 內文引用與參考文獻格式:「戶外遊憩研究」APA 7;造園景觀學會
- [ ] 內文結構:漏斗狀由大到小;主要架構為 IMRD

### 6.2 版面設定
- [ ] 邊界:上 2.54 cm、下 2.54 cm、左 3.17 cm、右 2.54 cm
- [ ] 行距:單行
- [ ] 頁碼:阿拉伯數字置於每頁下方正中間,由第一章第一節開始編號

### 6.3 封面與紙張
- [ ] 封面一律米色萊妮紙 200–250 磅(平裝本)
- [ ] 內頁至少 70 磅白色模造紙印刷
- [ ] 封面中間加東海浮水印

### 6.4 中文封面(段落固定行高 25pt;中文標楷體、英文 Times New Roman)
- [ ] 「東海大學/景觀學系/碩士論文 + Thesis for the Degree of Master / Department of Landscape Architecture / Tunghai University」:14 級
- [ ] 「指導教授:○○○博士 / Advisor:○○○ Ph.D.」:16 級
- [ ] 中文題目、英文題目:20 級
- [ ] 「研究生:○○○ / Graduate Student:○○○」:16 級
- [ ] 「中華民國○○年○月 + 西元年,月」:14 級

### 6.5 文稿排序
- [ ] 1. 中文封面
- [ ] 2. 論文指導教授及口試委員之簽名頁
- [ ] 3. 論文中、英文摘要(以不超過一頁為原則)
- [ ] 4. 誌謝辭或序言
- [ ] 5. 目錄、圖次、表次
- [ ] 6. 論文正文(格式依「造園景觀學報」)
- [ ] 7. 參考文獻(格式依「造園景觀學報」)
- [ ] 8. 附錄(含附表、附圖;含每篇引用文獻之原文摘錄,見 §5.3)
- [ ] 9. 封底
- [ ] 10. 書背:東海大學/景觀學系碩士(專)班/碩士論文/論文題目/畢業之中華民國年份加括弧如(116)而非學年/研究生:○○○ 撰

## 7. Claude Code 自動化(CLAUDE.md 規則摘要)

觸發方式:在筆電對 Claude Code 說「更新進度」或直接鍵入新進度,即執行:

1. **收件**:讀 `inbox/inbox.md` 全部未處理行。
2. **解析**:判斷每行屬於哪個專案/任務/文獻/書,更新對應 JSON(勾完成、寫進度百分比、補時間戳)。
3. **建檔**:新文獻 → 依 §5.2 模板建 `notes/literature/` 檔;新引用 → 提醒補 appendix-quotes。
4. **重算時程**:依完成狀況更新雙軌看板的剩餘天數與落後警示;落後項目列入「今天」頁置頂。
5. **檢查**:執行 §5.3 引用一致性檢查、format-checklist 未完成項統計。
6. **清空 inbox、commit、push** → GitHub Pages 自動部署 → 手機重新整理即見最新狀態。

Commit message 慣例:`progress: <摘要>`、`lit: <citekey>`、`checklist: <項目>`。

## 8. 資料、文檔歸類與工作流程建議

原則:**工具愈少愈好,每種資料只有一個家,全部免費。**

| 資料類型 | 放哪裡 | 理由 |
|---------|--------|------|
| 進度、任務、時程、檢核 | 本 app(repo 的 data/) | 唯一進度來源,手機桌面同步 |
| 文獻 PDF + 書目資料 | **Zotero**(免費,含 APA7 樣式) | 引用的唯一真實來源;裝「戶外遊憩研究」CSL 樣式;citekey 與本 repo 對齊 |
| 文獻筆記、附錄摘錄 | 本 repo `notes/`(**Obsidian 直接開此資料夾**) | Obsidian = 編輯器,repo = 儲存;Claude Code 可直接讀寫、檢查一致性 |
| 論文正文 | Word 檔(依系規定須用 Word),放 OneDrive/iCloud 或亦入 repo | 格式檢核以 §6 清單人工+Claude 比對 |
| 讀書筆記(30 本) | `notes/` 內簡短 md,一本一檔 | 進度勾在 books.json |
| Notion | **不建議使用** | 與 Obsidian/本 app 功能重疊,多一個要維護的地方 |

工作流程(日常):
1. 任何時候手機丟一句進度到輸入框。
2. 筆電開工時:`git pull` → 對 Claude Code 說「更新進度」→ 它整理一切。
3. 讀文獻:Zotero 收 PDF → Claude Code 建筆記模板 → Obsidian 填三問/十問 → 引用時補附錄摘錄。
4. 每週日看「時間軸」頁,確認雙軌口試各自剩餘天數,調整下週任務。
